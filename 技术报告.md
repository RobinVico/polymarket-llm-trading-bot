## 二、用户日常使用流程

### 2.1 完整流程图

```
[扫描] → [研究] → [入场] → [监控] → [重评] → [加仓 or 平仓]
  ↑                                              │
  └────────────── 闲时回到扫描 ──────────────────┘
```

### 2.2 入场流程 (新仓位)

| 步 | 在哪 | 做什么 |
|---|---|---|
| 1 | Dashboard 顶部 "🎯 复制给Claude" | 点按钮, DISCOVERY prompt 复制到剪贴板 |
| 2 | Claude.ai | 贴上, Research 模式跑, 等输出 |
| 3 | Polymarket 网页 | 按 Claude 建议手动买入 (bot 不下首单) |
| 4 | Dashboard 持仓表 | 找到新仓位, 填 q 输入框 (Claude 给的"校准后估算"%) |
| 5 | 信心下拉框 | 选 高/中/低, **改了自动保存** |
| 6 | 点 "保存" | 把 TP 存进 DB |

### 2.3 日常监控

- Dashboard 永远开着, 每 30s 自动刷新仓位/价格/盈亏
- 看 4 个 metric 卡: 总盈亏 / 持仓数 / 总投入+总收入 / 资产组合+现金
- 看 🎯 事件中心: 实时榜/涨跌榜/现值榜切换
- 看每个仓位的决策状态 badge:
  - 🟢 `HOLD` — 继续放着
  - 🟡 `MARGINAL` / `SOFT_NEGATIVE` — 需要重评
  - 🔴 `AT_TARGET` — 点 badge 走 sell
  - 灰 `PENDING` / `NO_META` — 还没填 q

  (v5.1 删除了 `FROZEN` / `FROZEN_FRESH` / `FROZEN_EXPIRED` / `SLOW_DROP` / `ABSOLUTE_FLOOR` 等 v5 止损相关状态, 见 §四.)

### 2.4 重评流程

| 步 | 在哪 | 做什么 |
|---|---|---|
| 1 | 仓位 "🔄 重评" 按钮 | 打开 reeval-panel |
| 2 | "📋 复制 prompt" | 复制 v5.2 prompt (含 Resolution 规则 + YES/NO 双向) |
| 3 | Claude.ai Research | Deep Research 8-30 min |
| 4 | Claude 输出 | hold / update_q={X}% / exit |
| 5 | Dashboard 选 | 按结果点 "维持 q" / 填新 q "更新 q" / "建议清仓" |

### 2.5 加仓 / 清仓 (v5 可直接在 dashboard 操作)

每个仓位的决策状态行末尾:

```
[$ 输入框] [＋加仓 绿]  [✗清仓 红]
```

- **加仓**: 输入美元金额 → 点 ＋加仓 → 弹 confirm "买 X 股 @ Y%, 约 $Z" → executor.buy() FAK
- **清仓**: 点 ✗清仓 → 弹 confirm "卖全 X 股 @ Y%" → executor.sell() 全平

**Bot 自动卖** 在 v5.1 这 4 个状态触发: `TAKE_PROFIT_PRICE` / `TAKE_PROFIT_PNL` / `STOP_LOSS` / `TIME_STOP`。其他都需用户确认。

### 2.6 紧急情况

- **服务挂了**: 跑 §12.2 重启流程
- **DB 损毁**: 跑 §12.3 灾难恢复
- **bot 自动卖出失败**: 看 `bot.log` 的 ERROR 行
- **强制立即备份**: `bash scripts/auto_backup.sh`

## 四、决策引擎 (v5.1 核心)

### 4.1 优先级 (第一个匹配胜出)

`modules/monitor.py:_evaluate_position()`:

| 优先级 | 状态 | 触发条件 (用什么价) | 动作 |
|---|---|---|---|
| 0 | `NO_META` | meta 行不存在 | 跳过 |
| 1a | `TAKE_PROFIT_PRICE` | **best_bid** ≥ 90¢ | **自动全卖** |
| 1b | `TAKE_PROFIT_PNL` | (**best_bid**−avg)/avg ≥ +100% | **自动全卖** |
| **2** | **`STOP_LOSS`** ⭐ v5.1 | **LLM 入场三档** (convergent -20% / hybrid -35% / event_driven 不止损只看 $0.05 地板; 未填 fallback -25%) | **自动全卖** |
| 3 | `TIME_STOP` | 距结算 ≤ 2 天 且 \|**cur** − entry\| < 5pp | **自动全卖** |
| 4 | Edge-based | `edge = q − **cur**` | 等用户确认 |

(止盈用 best_bid, 止损/TIME_STOP/edge 用 cur — 设计动机见下面"关键实现细节 (1)".)

**v5.1 删除的状态 (vs v5)**:
- `SLOW_DROP` / `FROZEN_FRESH` / `FROZEN` / `FROZEN_EXPIRED` — 整套急跌冻结机制基于 "亏了会涨回来" 的幻觉, 数据否决 (10% 概率), 全部删除。
- `ABSOLUTE_FLOOR` (40% 兜底) — 被 STOP_LOSS 25% 取代, 兜底永远到不了。
- `BLACKSWAN_HEDGE` (97% 减半) — 已被 TAKE_PROFIT_PRICE (90¢) 完全覆盖。
- `CONFIRMED_NEGATIVE` — 已合并入 `SOFT_NEGATIVE` 一档。
- DB 字段 `freeze_until` / `freeze_stop_price` / `last_q_update_with_negative_edge` 保留 (兼容老数据, monitor 不再读写)。

**v5.1 止盈规则设计动机**:
- **90¢ 价格止盈**: 90¢ 以上继续涨的边际收益很低 (年化约 25%), 但流动性变差 + UMA 结算风险变大。历史教训: Trump publicly insult 涨到 100¢ 没卖, 后被止损卖在 13¢。
- **+100% 浮盈翻倍止盈**: 翻倍是"已经赚到一倍"的心理/数学双重锚点, 后续回吐就是绝对损失。+70% 太早 (会过早锁住 Trump AI export +131%、Gemini 3.2 +103%、Hormuz 0-10 +76% 这类大赢家), +100% 更合理。
- **为什么放最高优先级**: 止盈是无条件的"该跑了"信号, 不应该被任何 edge 重评压住。

**v5.1 STOP_LOSS LLM 三档设计动机** (回测驱动, 5-22 ~ 5-23 数据迭代):

5 月初样本驱动出单一 -25% 阈值, 但 5-22 ~ 5-23 又验证了 6 个 STOP_LOSS 仓位, 卖错率 4/6 (67%):
- ❌ Senate reconciliation 卖在 $0.077 → 现在 $0.998 (+92pp)
- ❌ Malta turnout 卖在 $0.114 → 现在 $0.575 (+46pp)
- ❌ Israel airspace 卖在 $0.450 → 现在 $0.875 (+42pp)
- ❌ Caroline Elliott 卖在 $0.195 → 现在 $0.250 (+5pp)
- ✅ Backrooms 票房 卖在 $0.124 → 现在 $0.040 (-8pp, 卖对)
- ✅ Hormuz 20 ships 卖在 $0.501 → 现在 $0.310 (-19pp, 卖对)

卖对的是 "硬数据型" (票房/船运统计), 卖错的全是 "政治/谈判/外交型" — 价格震荡 ≠ 真相变化, 跌深后大反弹是常态。

据此引入 LLM 入场分类三档:
| 档 | 类型 | 阈值 |
|---|---|---|
| `convergent` | 真相收敛 (硬数据公布即定) | -20% |
| `hybrid` | 混合 (民调+政治, 如候选人选举) | -35% |
| `event_driven` | 事件驱动 (政治/外交/谈判) | 不止损, 仅 $0.05 地板兜底 |

分类来源: DISCOVERY prompt 让 Claude 根据 market resolution criteria 直接给出推荐档, 用户在 dashboard 入场流程选 dropdown。老仓位 / 未填 fallback 用 -25% (向后兼容)。

数据驱动反推: 如果当时这 6 个仓位有正确分类, 假设:
- Senate / Malta / Israel airspace 全是 event_driven → 不会触发 -25% 止损, 现在大反弹都能拿到。
- Backrooms / Hormuz 全是 convergent (-20%): Backrooms 跌过 -50% 也会触发, Hormuz 是 +5%(没亏)。
- Caroline 是 hybrid (-35%): -27% 不会触发, 现在 +5pp 可以拿。

预期: 改后总损失会从 $-12.15 减少到 $-2~3 (省 $9)。

**关键实现细节 (1): 触发用价不对称 — 止盈用 best_bid, 止损用 cur_price**

```python
# monitor.py _evaluate_position
best_bid = self.executor.get_best_bid(token_id)
bid_price = best_bid if best_bid is not None and best_bid > 0 else cur_price

# 止盈 (浮盈方向): 用 best_bid, 防 cur 虚高假触发
if bid_price >= TAKE_PROFIT_PRICE: ...
if (bid_price - avg) / avg >= TAKE_PROFIT_PNL_PCT: ...

# 止损 (浮亏方向): 用 cur_price, 防 best_bid 瞬时虚低误触发
if (avg - cur_price) / avg >= STOP_LOSS_PCT: ...
```

**为什么不对称** (Dell 案例 + 一般原理):
- **止盈陷阱**: Dell Q1 仓位 2026-05-21 cur=$0.905 但 orderbook bid 只有 $0.60 (spread 30pp+, 冷门财报市场). bot 看 cur 触发 TAKE_PROFIT_PRICE → sell 实际只能吃 $0.60 → 把"理论 +50%"锁成"实际 -29% 亏损". **这种错决策没法回头** (钱已经成交). 用 best_bid 触发: bid 没真到 90¢ → 不触发, 不会假胜利.
- **止损陷阱**: cur=$0.20 时 market maker 短暂撤单, bid 临时空只剩 $0.10. 用 bid 触发 STOP_LOSS → 卖在 $0.10 (最差点). 30 秒后 bid 回到 $0.18 (mm 回来挂单), 你白白卖在了瞬时最低. 用 cur 触发: cur 没真跌就不触发, 给市场恢复 spread 的时间.
- 同时也保持跟 -25% 回测甜蜜区数据一致 (回测用的是 `prices-history` 60min 聚合 mid 价, 跟 cur 同源, 跟 best_bid 不同源).
- `TIME_STOP` / Edge-based 也用 cur_price (settlement 临近判断 + qualitative 比较, 不是 execution 价).

**关键实现细节 (2): STOP_LOSS / TAKE_PROFIT_PNL 用 `pos.avg_price` 不用 `meta.entry_price`**

```python
# monitor.py _evaluate_position
avg = pos.get("avg_price") or entry or 0
# stop loss
if (avg - cur_price) / avg >= STOP_LOSS_PCT: ...
# take profit pnl
if (cur_price - avg) / avg >= TAKE_PROFIT_PNL_PCT: ...
```

**为什么不用 `meta["entry_price"]`** (历史踩过的坑):
- `entry_price` 是 db 里第一次入场的历史价, 加仓 / 二次入场后**不会更新**.
- 2026-05-19 发现 bug: Hormuz 20 ships 仓位 entry=$0.380 (5-14 第一次买), avg=$0.620 (5-19 二次入场), cur=$0.760. 老代码用 entry 算: (0.76-0.38)/0.38 = +100% → 误触发 TAKE_PROFIT_PNL. 实际真实浮盈仅 +22.6%.
- 5 月 30 仓位实际操作里 **0 次加仓 + 10 次二次入场** — 主要风险是二次入场.
- 改用 `avg_price` 多一行 fallback (`or entry or 0`), 同时防御加仓和二次入场两种场景.

**配套: 平仓后自动清理 `position_meta` 整行** (根治二次入场的 db 残留):

三个触发点:
1. `monitor.py:check_once()` 里 `executor.sell()` 成功后 → `clear_position_meta(token_id)`
2. `dashboard.py` 的 `/api/force_exit` 和 `/api/execute_state` 里 sell 成功后 → 同上
3. `monitor.py:check_once()` 心跳里对比 `polymarket positions ⊃ db meta`, 差集里的 token_id 全部清理 (覆盖用户在 polymarket app 直接卖、绕过 bot 的场景)

心跳清理的**关键守卫**: 只有 `positions 非空` 或 `(positions 空 AND cash > 0)` 时才执行 — 防止 API 失败 (返回 positions=[] AND cash=0) 时误删全部 meta. 实现见 `modules/db.py:clear_position_meta` 和 `get_all_meta_token_ids`.

#### Edge-based 子状态 (优先级 4 内)

## 六、DISCOVERY Prompt (新仓研究)

### 6.1 触发

Dashboard 点 "🤖 复制给Claude" → `/api/full_prompt` → DISCOVERY 模板 + 最近扫描结果。

### 6.2 关键流程

每个候选要 Claude:
1. **Bull / Bear 视角** (各 1-2 个权威来源)
2. **原始估算 raw**
3. **校准后** = `市场价 + 0.5 × (raw − 市场价)` (五折)
4. **校准后 Edge** = calibrated − cur
5. **执行成本** = taker 滑点 + 双边 2.2pp
6. **实际可获取 Edge** = 校准 edge − 执行成本

### 6.3 推荐门槛

| 模式 | 校准 edge 门槛 |
|---|---|
| 标准 | ≥ **8pp** |
| 中范围 | ≥ **11pp** |
| 大范围 | ≥ **14pp** |

### 6.4 价格区间不对称

| 模式 | 价格范围 |
|---|---|
| 标准 | 8% — 92% |
| 中 | 5% — 95% |
| 大 | 3% — 97% |

低端宽因 < 15% 是 longshot 反向操作甜区。

### 6.5 年化 IRR

```
IRR = (calibrated − p) / p × 365 / days × 100%
< 30%: 不推荐 | 30-100%: 可接受 | > 100%: 甜区
```

### v5.3 (2026-05-19) — 三 bug 一起修

**问题 1: STOP_LOSS / TAKE_PROFIT_PNL 用错了参考价**

- Bug: 公式用 `meta.entry_price` (第一次入场价), 但用户加仓 / 二次入场后 entry 不更新, 仅 `pos.avg_price` (polymarket 实时加权均价) 是对的.
- 数据证据: Hormuz 20 ships token 5月 BUY-SELL-BUY-SELL-BUY 三次, 每次都进新仓位, db.entry_price 永远停在 $0.380 (第一次), polymarket avg=$0.620, cur=$0.760. 老代码算 (0.76-0.38)/0.38 = +100% 误触 TAKE_PROFIT_PNL, 实际真实浮盈仅 +22.6%.
- 5 月数据: **0 次真加仓, 10 次二次入场** — 全是这个 bug.
- **修法**: monitor.py L80 加 `avg = pos.get("avg_price") or entry or 0`, 后面所有止盈止损用 `avg`.

**问题 2: portfolio_snapshot 写入了一行全零**

- Bug: `get_positions()` 失败返回 `[]`, `get_cash_balance()` 失败返回 `0.0`, monitor 把 `(total_value=0, cash=0)` 写入 db, 污染图表.
- **修法**: 加守卫 `if not positions and (cash is None or cash == 0): skip`.

**问题 3: db.position_meta 残留 (二次入场)**

- Bug: 用户卖光一个 token 后再买回来, db 里 entry_price / created_at / executed_action 仍然是老的, dashboard 显示错乱, executed_action 残留可能阻止下次自动卖.
- **修法**: 新增 `db.clear_position_meta(token_id)` 函数, 三个触发点:
  1. monitor `auto_sell` 成功后立即清理
  2. dashboard `/api/force_exit` + `/api/execute_state` 手动卖成功后清理
  3. monitor 心跳 sweep: 对比 `polymarket positions ⊃ db meta`, 差集 = 孤儿, 清掉
- 同时新增 `db.get_all_meta_token_ids()` helper

**关键设计决策固化** (写进 CLAUDE.md): "止盈止损用 avg_price 不用 entry_price", "平仓后整行删除 meta 而非清字段".

### v5.4 (2026-05-21) — 触发用价不对称: best_bid for TP, cur for SL

**问题: Dell 案例暴露 polymarket curPrice 是假象**

- Dell Q1 Infrastructure Solutions Group revenue 仓位 2026-05-21 19:53 触发 TAKE_PROFIT_PRICE
- polymarket 给 bot 的 curPrice = $0.905 (last trade 价, 偶发的一笔成交)
- 但 orderbook best_bid 只有 $0.60, best_ask $0.93, spread 30pp+ (冷门财报市场流动性差)
- bot sell FAK 8.33 股, makingAmount 6.11 股 (73% fill), 实际卖在 $0.60
- 期望 +50% 收益 → 实际 -29% 亏损 (锁个假胜利)

**改了**:
- executor.py 加 `get_best_bid(token_id)` (镜像 `get_best_ask`)
- monitor.py: TAKE_PROFIT_PRICE / TAKE_PROFIT_PNL 改用 `best_bid` 触发 (真能卖到才触发)
- best_bid 拉失败 fallback 回 cur_price (避免 bot 卡死)

**为什么 STOP_LOSS 仍用 cur_price** (非对称设计):
- 止损方向陷阱是"虚低" — bid 临时空了 (market maker 短暂撤单), 用 bid 触发会卖在最差成交价
- 止盈方向陷阱是"虚高" — last trade 偶发一笔, 用 cur 触发会以为锁了利润实际卖不到
- 数据连续性: -25% 阈值是基于 prices-history (mid 类) 算的, 止损用 cur 保持一致

**关键设计决策固化** (CLAUDE.md): "TAKE_PROFIT 用 best_bid, STOP_LOSS 用 cur, 这种不对称是有意的".

### v5.5 (2026-05-24) — 网络稳定性 / 守卫加强 / 曲线回填

**问题 1: 5-24 00:41 meta sweep 灾难**

- 当时 `data-api.polymarket.com` 挂 (No route to host), `clob.polymarket.com` 还活
- `get_positions()` 失败返回 `[]`, `get_cash_balance()` 成功返回 ~$50
- v5.3 守卫 `if positions or (cash is not None and cash > 0)` 被欺骗: `False or True = True` → sweep 执行
- `current_token_ids = set()`, `orphans = ALL 7 个 meta` → **全部 db.meta 被清光**
- 后果: 7 个有效仓位的 monitor_state 变 PENDING, bot 停止保护它们直到用户手动重填

**根因**: `positions=[]` 不能区分 "API 失败" vs "用户真无仓位"; `cash` API 跟 `positions` API 是独立 endpoint, 单边挂会绕过守卫.

**修了**:
- monitor.py sweep 守卫改成: `if positions:` (positions 必须非空才 sweep)
- 副作用: 用户真清光所有仓位时, 残留 meta 不自动清; 但残留无害 (monitor 只评估 polymarket 现存仓位), 新仓位时 sweep 会重新清

**问题 2: portfolio_snapshot 写入 cash=0 的 bug 行**

- `get_cash_balance` 失败返回 `0.0`, 跟"账户真没钱"长得一样
- v5.3 守卫只判 `not positions and cash == 0`, 漏了"positions 有 + cash=0" 这种 corner case
- 5-23 / 5-24 一共写了 5 行 `total_value=$30, cash=0, assets=$30` 的污染行 (实际 cash 是 $50, assets 应该是 $80)

**修了**:
- executor.py `get_cash_balance` 失败返回 `None` 而不是 `0.0` (语义清晰)
- monitor.py 守卫三道:
  1. `cash is None` 跳过 (cash API 失败)
  2. `positions=[] AND cash=0` 跳过 (两边都挂)
  3. `positions=[] AND db 有 meta 行` 跳过 (账户历史非空, positions 几乎肯定 API 挂)
- dashboard.py 两处 cash 用法加 `or 0` 兜底

**问题 3: 资产曲线长直线 + 长下线**

- 5-19 (改 sweep 重启那几次) + 5-23~24 (网络挂 + 守卫拒写) 留下两段大空白
- + 我之前删了 5 行污染数据后又添了空白

**修了** (一次性脚本, 不动代码):
- 找出空白段 (>1h 相邻 ts)
- 对每个 30min 补点 t:
  - cash(t) = baseline cash + 所有 t 之前 SELL revenue - 所有 t 之前 BUY cost
  - position(t) = 每个 token 的 BUY size sum - SELL size sum (filter: size > 0.1)
  - 排除已结算 token (用 prices-history 拉空作为信号, 比 gamma-api closed 更鲁棒)
  - price(t, token) = prices-history 最近 ≤t 的点
- 93 个补点插入, 5-24 16:00 衔接处偏差 $0.76 (vs 旧版 $7.35)

- `v4.db.bak_1779664774_step3_final` — 最终 (706 行)

### 15.2 设计原则

1. **自己用零摩擦** — 本机 (Mac mini) 不需要登录
2. **手机/笔记本用首次登录, 之后 90 天免** — 不能每次都输密码
3. **公网无 URL 可攻击** — 即使有人知道你的项目存在也找不到入口
4. **设计假设公开** — 安全靠 secret 不靠 obscurity (Kerckhoffs)

### 15.3 三层防御架构

```
[外部公网]    ─ ✗ ─→ (无路径; DNS 在 tailnet 外不解析)
[Tailnet 设备] ─→ tailscale serve (本机 daemon, HTTPS 终结)
                     │
                     ▼
                Flask (127.0.0.1:5051, 只 listen localhost)
                     │
                     ├─ 来源判定:
                     │    A. remote_addr ∈ {127.0.0.1, ::1} AND 无 X-Forwarded-For
                     │       → 本机直连, 放行 (零摩擦)
                     │    B. 其他来源 → 走身份验证
                     │
                     ▼
                Flask session 检查
                     │
                     ├─ session.authed=True   → 放行
                     ├─ path=/login           → 放行
                     ├─ method=POST 且未授权  → 401 JSON
                     └─ GET 且未授权          → 302 → /login
```

**关键踩坑** (5-29 实测确认): Tailscale serve 是 daemon 内部反代, **它转发到 Flask 时 `remote_addr` 也是 `127.0.0.1`**. 如果只判 `remote_addr in (127.0.0.1, ::1)` 就放行, **整层鉴权完全失效, tailnet/Funnel 流量被当成本机**. 必须同时判 `X-Forwarded-For` 头存在与否 — Tailscale serve 反代会塞这个头, 本机直连不会有.

### 15.4 Tailscale 层 (基础设施, 一行命令切换)

| 模式 | 命令 | 谁能访问 |
|---|---|---|
| Funnel (公网) | `tailscale funnel --bg on` | 任何拿到 URL 的人 |
| Serve (tailnet-only) | `tailscale serve --bg http://localhost:5051` | 只接入 tailnet 的设备 |

**默认推荐 Serve**. URL 不变 (仍是 `https://<hostname>.tailXXXXXX.ts.net`), 但 DNS / TLS 只对 tailnet 设备 resolve, 公网完全找不到入口.

**临时分享给非 tailnet 朋友**: 临时 `tailscale funnel --bg on`, 朋友凭密码看, 看完 `tailscale funnel --bg off`. 即使 Funnel 开着, Flask 密码层仍挡住攻击.

### 15.5 Flask 密码层 (代码, ~80 行集中在 `modules/dashboard.py`)

**配置块** (放在 `create_app()` 一进来):
```python
app.secret_key = os.environ.get("FLASK_SECRET_KEY") or secrets.token_urlsafe(32)
app.config.update(
    PERMANENT_SESSION_LIFETIME=timedelta(days=90),
    SESSION_COOKIE_HTTPONLY=True,   # JS 偷不到
    SESSION_COOKIE_SAMESITE="Lax",  # CSRF 防御
    SESSION_COOKIE_SECURE=True,     # 只走 HTTPS (Tailscale 都是 HTTPS)
)
```

**中间件** (`@app.before_request`):
```python
is_local_direct = (
    flask_request.remote_addr in ("127.0.0.1", "::1")
    and not flask_request.headers.get("X-Forwarded-For")
)
if is_local_direct:
    return None
if session.get("authed"):
    return None
if flask_request.path in ("/login", "/favicon.ico"):
    return None
if flask_request.method == "POST":
    return jsonify({"ok": False, "error": "auth required"}), 401
return redirect("/login")
```

**登录路由** (`/login`):
- GET → 渲染极简 LOGIN_HTML form
- POST → 比对 `os.environ["DASHBOARD_PASSWORD"]`, 正确则 `session["authed"]=True; session.permanent=True`

**限流** (in-memory dict): 5 次错密码 → 30 分钟 lock by IP. 简单可靠, 不依赖 Flask-Limiter.

### 15.6 端口收紧 (`main.py`)

```python
app.run(host="127.0.0.1", port=5051, debug=False)  # 不是 0.0.0.0
```

防御深度: 即使 Tailscale daemon 出 bug, Flask 也不会暴露在 LAN/Wi-Fi 端口扫描可见.

### 15.10 安全等级 / 攻击面分析

要攻入 dashboard 需要同时打穿三关:

| 关卡 | 攻击成本 |
|---|---|
| 进 tailnet | 拿到 Tailscale 账号 (邮箱 + 密码 + 2FA), 难度高 |
| Flask 密码 | 暴力破解 32 字符随机, 理论上 ~2^192 次, 不可能 |
| 限流 | 5 次失败 30 分 lock by IP, 自动化爆破成本巨高 |

**剩余攻击面**:
- secret_key 泄漏 → 攻击者可伪造 session (绕过密码). 缓解: secret_key 在 gitignored `.env`, 公开文档绝不写真值.
- Tailscale 账号被偷 → 完全绕过. 缓解: 启用 2FA, 用密码管理器.
- in-memory 限流重启清零 → 小漏洞, 但攻击者要先重启系统才能利用, 实际不可行.

### 15.11 公开本设计的边界

本章可以完整发布 — **架构 / 代码骨架 / 部署步骤公开不影响安全** (Kerckhoffs principle).

**绝对不能写进公开文档的真值** (即使是私有 README 也建议放占位符):

- `DASHBOARD_PASSWORD` 真值
- `FLASK_SECRET_KEY` 真值
- 真实 Tailscale 账号邮箱
- proxy wallet 地址 (公开 = 任何人能查链上完整持仓和交易历史)
- private key (永远不要)
- 真实 tailnet hostname (可保守替换为 `<hostname>.tailXXXXXX.ts.net`)

## 十九、v5.9 仓位 sizing 公式 + Cluster 相关性簇 (2026-06-01)

> 用户原话: "我现在每个仓位都是固定 5 美金, 低概率题手动减到 2-3, 这都是凭感觉, 没数学依据".
> Claude.ai Research 模式产出了完整研究报告 (Kelly + LLM 折扣 + cluster cap + DD 预算), v5.9 落地为生产代码.

### 19.1 Research 报告关键发现

1. **三个伊朗仓位是同一 cluster** (相关性几乎 1.0). 按现价口径 $21.07 已经超过 20% bankroll cap $17.29, **绝不能再开第 4 个伊朗仓**.
2. **us-iran-peace 仓 edge 最大**, 应从 $5 加到 ~$8.
3. **Mazzei 仓 q=0.87 + p=0.131 触发 sanity check** (74pp edge 不真实), 数据可疑.

### 19.2 公式设计 (`modules/sizing.py`)

5 步 (单层折扣方案 — 用户决策):

1. **edge 检查**: edge = q − p; ≤0 return $0
2. **单 token Kelly**: kelly_f = edge / (1−p); raw = bankroll × kelly_f × 0.25 (1/4 Kelly)
3. **days + longshot 调整**: days_factor = sqrt(21/days) clip [0.40, 1.0]; longshot_mult = 1.0 if p≥0.15 else 0.5+0.5×(p/0.15)
4. **cluster + DD budget clip**: cluster_room = cap − cur_exp; dd_cap = (30 − exposed_dd) / TIER_DD[tier]; clip 取 min
5. **hard bound**: clip 到 [$1, $15]

**为什么单层折扣**: DISCOVERY prompt 已经做 q' = p + 0.5×(q_raw − p) 收缩. 公式再做一层 H 折扣会让总信任度 = 0.5×H = 0.375 (high). 用户决定: 公式不再二次折扣, q 直接用 DISCOVERY 输出. 总信任度 = 0.5. confidence 字段接收但不进公式.

### 19.3 关键设计: 现价口径 (用户审核 ①)

`cluster_exposure_usd` 和 `portfolio_exposed_dd_usd` **都用 cur_price × size 现价**, 不是 cost_basis 入场成本.

**否则会自相矛盾**: bankroll 是 `cash + Σ(cur_price × size)` 现价口径, 如果 cluster_exposure 用 cost_basis, 实测当前 3 个伊朗仓 cost $15 < cap $17.29 → 公式认为"还有空间" → 会错误推荐第 4 个伊朗仓 $2.29. 而现价 $21.07 > cap → 正确拒绝.

### 19.4 Cluster 命名约定 `<topic>-<direction>` (用户反馈)

Cluster 不是按话题分, 是按**相关性**分:
> 如果此仓位赢了, 其他哪些仓位也倾向于一起赢? 一起赢 → 同 cluster

例:
- 3 个伊朗 No (押"不缓和") → 同 cluster `iran-deescalation-no`
- 1 个 No + 1 个 Yes 同话题 → **不同 cluster** (反相关!)
- SCOTUS 保守裁决 + 共和党中期赢 → 可能同 `gop-conservative-momentum-win` (跨话题相关)

### 19.12 LANGUAGE LOCK — 强制中文输出 (2026-06-03 补丁)

**用户问**: "新仓位 AI 分析时点了 skill, 但 Claude 总是回复英文报告, 为什么?"

1. `modules/dashboard.py:/api/full_prompt` 在最顶部 prepend 一个"LANGUAGE LOCK"段:

### 20.5 Resolution 检测机制

**新建 `modules/resolution_check.py`**:

```python
check_resolution(token_id, side=None)
# → None  (未结算 / 查询失败)
# → {"final_outcome": 1 or 0, "resolved_at": ISO}

update_unresolved_closed_positions(limit=50)
# → (checked, updated)
```

**调用入口**:
- `monitor.run_loop` 每 `RESOLUTION_CHECK_ROUNDS=120` 轮跑一次 (30s × 120 = 1 小时), 不阻塞主决策.
- `scripts/backfill_closed_resolution.py` 一次性回填老 closed_positions.

**`final_outcome` 语义 (关键!)**:
我们存的不是 "Yes 的最终概率", 是 "**持有 side 那一边**的最终概率". 这样 `is_correct = (final_outcome >= 0.5)` 永远对.

例:
- Yes 仓位 + 市场 Yes 中 → outcomePrices[Yes_idx]=1.0 → final_outcome=1, is_correct=1 (赌赢)
- Yes 仓位 + 市场 No 中  → outcomePrices[Yes_idx]=0.0 → final_outcome=0, is_correct=0 (赌输)
- No  仓位 + 市场 No 中  → outcomePrices[No_idx] =1.0 → final_outcome=1, is_correct=1 (赌赢)
- No  仓位 + 市场 Yes 中 → outcomePrices[No_idx] =0.0 → final_outcome=0, is_correct=0 (赌输)

**判断 token_id 对应的 outcome index**: 优先用 `clobTokenIds.index(token_id)`, fallback 用 `side` 推断 (Gamma 默认 outcomes=["Yes","No"], index 0=Yes, 1=No).

**幂等设计**: `update_closed_resolution(...)` 带 `WHERE is_resolved=0` 守卫, 同一行被多次调用不会重复 UPDATE.

### 20.6 Tag backfill 脚本

**问题**: 老 closed_positions 没记 tag (v5.10 之前 schema 没这字段).

**新建 `scripts/backfill_closed_tag.py`**:

- `fetch_tag_for_slug`: 先 `events?slug=<market_slug>` 命中 events.tags. 失败 fallback `markets?slug=...` 取 events[0].id 再 `events?id=...`. (Polymarket tags 字段在 events 端点, 不在 markets 端点.)

## 二十一、v5.10.2 /history 数据正确性大修 + DNS 污染兜底 (2026-06-12)

> 起因: 用户反馈 /history 页面"一塌糊涂、看不懂、数据不准". 自检发现**远不止 UI 问题** — 三个数据层 bug 让整页统计基本是错的, 外加本机 DNS 污染让现价/结算检测全部失效.

### 21.1 Bug 1: No 仓 PnL 符号全反 (severity: 高)

`db.save_closed_position()` 对 No 仓用 `(avg - exit) × size`, 以为价格是"Yes 口径". 实际三个 caller (monitor.auto_sell / force_exit / execute_state) 传入的 `pos.avg_price` / `cur_price` 都是 **持有 token 自己的价格** (No 仓传的就是 No token 价), 正确公式不分 side 永远是 `(exit - avg) × size`.

**症状特征** (一眼可査): TAKE_PROFIT_PRICE (No token 卖在 $0.90) 显示 -75% 亏损; STOP_LOSS 显示 +58% 盈利. 完全自相矛盾.

**影响范围**: 9 行 (非 BACKFILL 的 No 仓). BACKFILL_FROM_TRADES 行从 trade 数据直接算, 本来就对.
**修复**: 公式改对 + `scripts/migrate_v5_10_2.py` 翻转历史 9 行 (幂等, migrations 哨兵表防重复跑, 自动备份 v4.db).

### 21.2 Bug 2: No 仓 is_correct 全反 (severity: 高)

`resolution_check.check_resolution()` 返回的 `final_outcome` 已经是**持有 side 的最终概率** (按持有 token 在 clobTokenIds 里的 index 取 outcomePrices), 文档写明 is_correct 应直接 `>= 0.5`. 但 `db.update_closed_resolution()` 把它当"Yes 的概率"对 No 仓二次翻转 → 已结算 No 仓胜负全反.

**影响**: 修正 7 行. 已结算胜率从 3/13 (23%) 修正为 8/13 (62%) — 之前页面把大量赌赢的 No 仓显示成赌输.

### 21.3 Bug 3: Gamma /markets 默认过滤 closed 市场 → resolution 检测永远空转 (severity: 高)

cron 连跑数周 `checked=50 updated=0` 无一行报错. 根因: **Gamma `/markets` 端点默认不返回 `closed=true` 的市场** — 而 resolution 检查找的恰恰就是已关闭市场. `markets?clob_token_ids=X` 对已结算市场永远返回 `[]`, 老的 events fallback 又拿市场 slug 去查 event slug (多市场打包 event 如韩国地方选举对不上) 也空. 之前侥幸结算的 13 行全靠"closed=False 但价格 ≥0.99"的软结算路径.

**修复**: `_fetch_market_any()` 五档查询链 (命中即停):
1. `markets?clob_token_ids+closed=true` ← 正式结算主路径 (新)
2. `markets?clob_token_ids` ← 还开着的市场, 软结算判定
3. `markets?slug+closed=true` (新)
4. `markets?slug` (新)
5. `events?slug` 反查 (原有)

slug 路径都校验 `token_id ∈ clobTokenIds` 防串市场. **效果: 一轮回填 56 token 结算成功 35 个**, 剩 21 个全是未到期市场 (6/30 / 9/30 / 世界杯).

附带修复: `get_unresolved_closed_positions()` 按 token_id GROUP BY 去重 + limit 50→100 (旧版 `ORDER BY exit_at DESC LIMIT 50` 让最老的 token 在积压 >50 时**永远轮不到检查**).

### 21.4 本机 DNS 污染 + 进程级 DoH guard

## 二十七、v7.0 · 出场策略重设计 (修"卖飞赢家" + "绕一圈回原点") (2026-06-22)

> 背景 (复盘): 6/8 后 16 笔净实现 ≈ **−$0.12** (止盈赚的几乎全被砍仓亏的抵消)。两个事件型仓 (#79 伊朗关领空 −54%、#86 JD Vance −43%) 被"重评建议 exit"砍在坑底, 结算却都赢, 合计少赚 ~$16 (账户 ~20%)。理论佐证: Kaminski & Lo (2014) —— 均值回复/负序列相关市场里按现价止损是负期望; Snowberg & Wolfers (2010) favorite-longshot 偏差。用户选范围 = **根因 + 出场机制 (不含自动加仓)**。

### 根因①: 重评 q 反锚定 (喂"大跌前价格中枢")
- 旧问题: 重评只把 被砸现价 + 入场价 + 原 q + 新闻 喂 AI, **从不喂"砸盘前中枢"** → AI 顺低价给低 q → 微负 edge → 建议 exit → 坑底割肉。
- `auto_reeval._pre_dump_center(token_id, cur, hist=None)`: 用现成 `executor.get_prices_history(interval='max', fidelity='60')` 取 trailing 窗口、**排除最近 `CENTER_SKIP_H`(默认6h, 即大跌段)**、取中位数; 数据不足 / `|center−cur|<0.01` → None; **永不抛异常**。
- 穿线 (最小改动, 不动编排签名): `run_and_store` 算一次挂 `pos['_pre_dump_center']`(给 prompt) + `d['_pre_dump_center']/d['_price_curve_json']`(落库); `_build_prompt` 读 pos 上 center → `build_reeval_prompt(pre_dump_center=)` + 指令尾拼一行"以大跌前中枢为参照、勿锚现价"。GLM + Claude 两路都过 `_build_prompt` → 自动一致。
- prompt: `REEVAL_PROMPT` 盲评块把现价重述为"大跌后被压低价" + 新增"大跌前价格中枢(盲评以此为参照)"行 (`_to_yes_no`); **`.format` 新 key 无条件传** (None 时 fallback 空串, 防 KeyError —— 手动 `/api/reeval_prompt` 也用它)。

### 根因②: event_driven exit 护栏
- `auto_reeval.guard_event_driven_exit(action, decision, tier, cur) -> (action_eff, downgraded, why)` (纯函数, 不碰 DB/不卖): event_driven 的 exit 仅在 `thesis_broken` 或 `(new_q − cur) ≤ −EXIT_GUARD_EDGE`(默认 0.08) 才放行; 否则降级 `update_q`(应用 new_q 继续持有)。**默认安全**: new_q 缺失/越界且非 thesis_broken → 持有。
- 两处调用 (卖之前、claim 之后): 离线 `_auto_execute` + 在线 `dashboard.auto_reeval_confirm`; 都取新鲜 `get_position_meta().stop_loss_tier`; 在线 update_q 返回带 `why`。

### 出场③: 止盈分档 + 分批卖一半
- `_evaluate_position` 1a 按 tier 分: event_driven best_bid ≥ `TAKE_PROFIT_PRICE_EVENT_DRIVEN`(0.92) → **卖一半** (`partial:True`, flag `tp_half_sold`); convergent 且 ≤`TAKE_PROFIT_CONVERGENT_NEAR_DAYS`(3)天 → 0.88 全卖; 其余 → 0.90 全卖。**event_driven 不走 0.90 通用全卖** (否则留下的半仓立刻被卖, "留一半跑"就废)。
- `check_once` partial 分支: 卖 size×0.5 成功 → mark `tp_half_sold` + `save_closed_position(size=半, exit_reason='TAKE_PROFIT_HALF')` + **绝不 clear_position_meta** (余量继续被管, size 下拍由 get_positions 自动减半) + flag 防重复; 失败 → 不记 flag/不清 meta 下拍重试。

### 出场④: 收敛型移动止损 + 确认
- `position_meta.peak_price` (新列, NULL-tolerant); `check_once` 每心跳 `max(old, cur)` 更新 (写 DB + 改内存 meta, 仿 entry 自愈)。
- convergent 止损改: 从 peak 回撤 ≥ `TRAILING_STOP_PCT_CONVERGENT`(0.20; ≤3天 `_NEAR` 0.12); `self._trail_breach[token]` 连续计数 ≥ `TRAILING_CONFIRM_ROUNDS`(6) 拍才算 (恢复即清零) → 走现有 PENDING_REEVAL/硬卖。hybrid/legacy 维持入场锚; event_driven 不变 (只地板)。
- **`_maybe_trigger_auto_reeval` 关键修**: `state=='PENDING_REEVAL'` 跳过 entry 锚 gate —— 否则收敛型在 (相对入场仍盈利时) 的移动止损会卡死 (既不卖也不触发重评)。确认计数用内存 = 重启清零只延迟不误触发; peak 在 DB 持久, PENDING_REEVAL 是持久后果态。

### 出场⑤: 记数据 (校准用)
- `auto_reeval_suggestions` 加 `pre_dump_center REAL` + `price_curve TEXT`; `update_auto_reeval_decision` 落 (cur_price 触发时已存 + new_q + center + 曲线 → 三元组同行)。`SELECT *` 自动进 dashboard, 无需改路由。

### 契约 / 不要随意改回
- partial / 卖失败 **绝不 clear_position_meta** (只全量成功才清)。护栏默认安全 = 不确定就持有。$0.05 地板永远第一、无条件。确认计数**用内存**, 不持久化"首次跌破时间戳"(长停机重启会立即误卖)。center 抓取失败 → None 重评照常。`build_reeval_prompt` 新 `.format` key 无条件传。stop 路径用 `is_configured()`(非 is_enabled), 急停也只冻结不盲卖。**event_driven 不走 0.90 通用全卖**。
- env: `AUTO_REEVAL_CENTER_WINDOW_H`(24)/`CENTER_SKIP_H`(6)/`EXIT_GUARD_EDGE`(0.08); monitor 常量 `TAKE_PROFIT_PRICE_EVENT_DRIVEN`(0.92)/`TAKE_PROFIT_HALF_FRACTION`(0.5)/`TAKE_PROFIT_PRICE_CONVERGENT_NEAR`(0.88)/`TAKE_PROFIT_CONVERGENT_NEAR_DAYS`(3)/`TRAILING_STOP_PCT_CONVERGENT`(0.20)/`_NEAR`(0.12)/`TRAILING_CONFIRM_ROUNDS`(6)。
- ⚠️ 阈值是 51 笔小样本**低置信度方向值**, 先观察 + 用 ⑤ 攒的真实数据校准再调。

### 验证
