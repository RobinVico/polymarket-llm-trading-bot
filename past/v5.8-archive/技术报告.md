## 二、用户日常使用流程

### 2.1 完整流程图

```
[扫描] → [研究] → [入场] → [监控] → [重评] → [加仓 or 平仓]
  ↑                                              │
  └────────────── 闲时回到扫描 ──────────────────┘
```

### 2.2 入场流程 (新仓位)

| 步 | 在哪 | 做什么 |
|---|---|---|
| 1 | Dashboard 顶部 "🎯 复制给Claude" | 点按钮, DISCOVERY prompt 复制到剪贴板 |
| 2 | Claude.ai | 贴上, Research 模式跑, 等输出 |
| 3 | Polymarket 网页 | 按 Claude 建议手动买入 (bot 不下首单) |
| 4 | Dashboard 持仓表 | 找到新仓位, 填 q 输入框 (Claude 给的"校准后估算"%) |
| 5 | 信心下拉框 | 选 高/中/低, **改了自动保存** |
| 6 | 点 "保存" | 把 TP 存进 DB |

### 2.3 日常监控

- Dashboard 永远开着, 每 30s 自动刷新仓位/价格/盈亏
- 看 4 个 metric 卡: 总盈亏 / 持仓数 / 总投入+总收入 / 资产组合+现金
- 看 🎯 事件中心: 实时榜/涨跌榜/现值榜切换
- 看每个仓位的决策状态 badge:
  - 🟢 `HOLD` — 继续放着
  - 🟡 `MARGINAL` / `SOFT_NEGATIVE` — 需要重评
  - 🔴 `AT_TARGET` — 点 badge 走 sell
  - 灰 `PENDING` / `NO_META` — 还没填 q

  (v5.1 删除了 `FROZEN` / `FROZEN_FRESH` / `FROZEN_EXPIRED` / `SLOW_DROP` / `ABSOLUTE_FLOOR` 等 v5 止损相关状态, 见 §四.)

### 2.4 重评流程

| 步 | 在哪 | 做什么 |
|---|---|---|
| 1 | 仓位 "🔄 重评" 按钮 | 打开 reeval-panel |
| 2 | "📋 复制 prompt" | 复制 v5.2 prompt (含 Resolution 规则 + YES/NO 双向) |
| 3 | Claude.ai Research | Deep Research 8-30 min |
| 4 | Claude 输出 | hold / update_q={X}% / exit |
| 5 | Dashboard 选 | 按结果点 "维持 q" / 填新 q "更新 q" / "建议清仓" |

### 2.5 加仓 / 清仓 (v5 可直接在 dashboard 操作)

每个仓位的决策状态行末尾:

```
[$ 输入框] [＋加仓 绿]  [✗清仓 红]
```

- **加仓**: 输入美元金额 → 点 ＋加仓 → 弹 confirm "买 X 股 @ Y%, 约 $Z" → executor.buy() FAK
- **清仓**: 点 ✗清仓 → 弹 confirm "卖全 X 股 @ Y%" → executor.sell() 全平

**Bot 自动卖** 在 v5.1 这 4 个状态触发: `TAKE_PROFIT_PRICE` / `TAKE_PROFIT_PNL` / `STOP_LOSS` / `TIME_STOP`。其他都需用户确认。

- **服务挂了**: 跑 §12.2 重启流程
- **DB 损毁**: 跑 §12.3 灾难恢复
- **bot 自动卖出失败**: 看 `bot.log` 的 ERROR 行

## 四、决策引擎 (v5.1 核心)

### 4.1 优先级 (第一个匹配胜出)

`modules/monitor.py:_evaluate_position()`:

| 优先级 | 状态 | 触发条件 (用什么价) | 动作 |
|---|---|---|---|
| 0 | `NO_META` | meta 行不存在 | 跳过 |
| 1a | `TAKE_PROFIT_PRICE` | **best_bid** ≥ 90¢ | **自动全卖** |
| 1b | `TAKE_PROFIT_PNL` | (**best_bid**−avg)/avg ≥ +100% | **自动全卖** |
| **2** | **`STOP_LOSS`** ⭐ v5.1 | **LLM 入场三档** (convergent -20% / hybrid -35% / event_driven 不止损只看 $0.05 地板; 未填 fallback -25%) | **自动全卖** |
| 3 | `TIME_STOP` | 距结算 ≤ 2 天 且 \|**cur** − entry\| < 5pp | **自动全卖** |
| 4 | Edge-based | `edge = q − **cur**` | 等用户确认 |

(止盈用 best_bid, 止损/TIME_STOP/edge 用 cur — 设计动机见下面"关键实现细节 (1)".)

**v5.1 删除的状态 (vs v5)**:
- `SLOW_DROP` / `FROZEN_FRESH` / `FROZEN` / `FROZEN_EXPIRED` — 整套急跌冻结机制基于 "亏了会涨回来" 的幻觉, 数据否决 (10% 概率), 全部删除。
- `ABSOLUTE_FLOOR` (40% 兜底) — 被 STOP_LOSS 25% 取代, 兜底永远到不了。
- `BLACKSWAN_HEDGE` (97% 减半) — 已被 TAKE_PROFIT_PRICE (90¢) 完全覆盖。
- `CONFIRMED_NEGATIVE` — 已合并入 `SOFT_NEGATIVE` 一档。
- DB 字段 `freeze_until` / `freeze_stop_price` / `last_q_update_with_negative_edge` 保留 (兼容老数据, monitor 不再读写)。

**v5.1 止盈规则设计动机**:
- **90¢ 价格止盈**: 90¢ 以上继续涨的边际收益很低 (年化约 25%), 但流动性变差 + UMA 结算风险变大。历史教训: Trump publicly insult 涨到 100¢ 没卖, 后被止损卖在 13¢。
- **+100% 浮盈翻倍止盈**: 翻倍是"已经赚到一倍"的心理/数学双重锚点, 后续回吐就是绝对损失。+70% 太早 (会过早锁住 Trump AI export +131%、Gemini 3.2 +103%、Hormuz 0-10 +76% 这类大赢家), +100% 更合理。
- **为什么放最高优先级**: 止盈是无条件的"该跑了"信号, 不应该被任何 edge 重评压住。

**v5.1 STOP_LOSS LLM 三档设计动机** (回测驱动, 5-22 ~ 5-23 数据迭代):

5 月初样本驱动出单一 -25% 阈值, 但 5-22 ~ 5-23 又验证了 6 个 STOP_LOSS 仓位, 卖错率 4/6 (67%):
- ❌ Senate reconciliation 卖在 $0.077 → 现在 $0.998 (+92pp)
- ❌ Malta turnout 卖在 $0.114 → 现在 $0.575 (+46pp)
- ❌ Israel airspace 卖在 $0.450 → 现在 $0.875 (+42pp)
- ❌ Caroline Elliott 卖在 $0.195 → 现在 $0.250 (+5pp)
- ✅ Backrooms 票房 卖在 $0.124 → 现在 $0.040 (-8pp, 卖对)
- ✅ Hormuz 20 ships 卖在 $0.501 → 现在 $0.310 (-19pp, 卖对)

卖对的是 "硬数据型" (票房/船运统计), 卖错的全是 "政治/谈判/外交型" — 价格震荡 ≠ 真相变化, 跌深后大反弹是常态。

据此引入 LLM 入场分类三档:
| 档 | 类型 | 阈值 |
|---|---|---|
| `convergent` | 真相收敛 (硬数据公布即定) | -20% |
| `hybrid` | 混合 (民调+政治, 如候选人选举) | -35% |
| `event_driven` | 事件驱动 (政治/外交/谈判) | 不止损, 仅 $0.05 地板兜底 |

分类来源: DISCOVERY prompt 让 Claude 根据 market resolution criteria 直接给出推荐档, 用户在 dashboard 入场流程选 dropdown。老仓位 / 未填 fallback 用 -25% (向后兼容)。

数据驱动反推: 如果当时这 6 个仓位有正确分类, 假设:
- Senate / Malta / Israel airspace 全是 event_driven → 不会触发 -25% 止损, 现在大反弹都能拿到。
- Backrooms / Hormuz 全是 convergent (-20%): Backrooms 跌过 -50% 也会触发, Hormuz 是 +5%(没亏)。
- Caroline 是 hybrid (-35%): -27% 不会触发, 现在 +5pp 可以拿。

预期: 改后总损失会从 $-12.15 减少到 $-2~3 (省 $9)。

**关键实现细节 (1): 触发用价不对称 — 止盈用 best_bid, 止损用 cur_price**

```python
# monitor.py _evaluate_position
best_bid = self.executor.get_best_bid(token_id)
bid_price = best_bid if best_bid is not None and best_bid > 0 else cur_price

# 止盈 (浮盈方向): 用 best_bid, 防 cur 虚高假触发
if bid_price >= TAKE_PROFIT_PRICE: ...
if (bid_price - avg) / avg >= TAKE_PROFIT_PNL_PCT: ...

# 止损 (浮亏方向): 用 cur_price, 防 best_bid 瞬时虚低误触发
if (avg - cur_price) / avg >= STOP_LOSS_PCT: ...
```

**为什么不对称** (Dell 案例 + 一般原理):
- **止盈陷阱**: Dell Q1 仓位 2026-05-21 cur=$0.905 但 orderbook bid 只有 $0.60 (spread 30pp+, 冷门财报市场). bot 看 cur 触发 TAKE_PROFIT_PRICE → sell 实际只能吃 $0.60 → 把"理论 +50%"锁成"实际 -29% 亏损". **这种错决策没法回头** (钱已经成交). 用 best_bid 触发: bid 没真到 90¢ → 不触发, 不会假胜利.
- **止损陷阱**: cur=$0.20 时 market maker 短暂撤单, bid 临时空只剩 $0.10. 用 bid 触发 STOP_LOSS → 卖在 $0.10 (最差点). 30 秒后 bid 回到 $0.18 (mm 回来挂单), 你白白卖在了瞬时最低. 用 cur 触发: cur 没真跌就不触发, 给市场恢复 spread 的时间.
- 同时也保持跟 -25% 回测甜蜜区数据一致 (回测用的是 `prices-history` 60min 聚合 mid 价, 跟 cur 同源, 跟 best_bid 不同源).
- `TIME_STOP` / Edge-based 也用 cur_price (settlement 临近判断 + qualitative 比较, 不是 execution 价).

**关键实现细节 (2): STOP_LOSS / TAKE_PROFIT_PNL 用 `pos.avg_price` 不用 `meta.entry_price`**

```python
# monitor.py _evaluate_position
avg = pos.get("avg_price") or entry or 0
# stop loss
if (avg - cur_price) / avg >= STOP_LOSS_PCT: ...
# take profit pnl
if (cur_price - avg) / avg >= TAKE_PROFIT_PNL_PCT: ...
```

**为什么不用 `meta["entry_price"]`** (历史踩过的坑):
- `entry_price` 是 db 里第一次入场的历史价, 加仓 / 二次入场后**不会更新**.
- 2026-05-19 发现 bug: Hormuz 20 ships 仓位 entry=$0.380 (5-14 第一次买), avg=$0.620 (5-19 二次入场), cur=$0.760. 老代码用 entry 算: (0.76-0.38)/0.38 = +100% → 误触发 TAKE_PROFIT_PNL. 实际真实浮盈仅 +22.6%.
- 5 月 30 仓位实际操作里 **0 次加仓 + 10 次二次入场** — 主要风险是二次入场.
- 改用 `avg_price` 多一行 fallback (`or entry or 0`), 同时防御加仓和二次入场两种场景.

**配套: 平仓后自动清理 `position_meta` 整行** (根治二次入场的 db 残留):

三个触发点:
1. `monitor.py:check_once()` 里 `executor.sell()` 成功后 → `clear_position_meta(token_id)`
2. `dashboard.py` 的 `/api/force_exit` 和 `/api/execute_state` 里 sell 成功后 → 同上
3. `monitor.py:check_once()` 心跳里对比 `polymarket positions ⊃ db meta`, 差集里的 token_id 全部清理 (覆盖用户在 polymarket app 直接卖、绕过 bot 的场景)

心跳清理的**关键守卫**: 只有 `positions 非空` 或 `(positions 空 AND cash > 0)` 时才执行 — 防止 API 失败 (返回 positions=[] AND cash=0) 时误删全部 meta. 实现见 `modules/db.py:clear_position_meta` 和 `get_all_meta_token_ids`.

#### Edge-based 子状态 (优先级 4 内)

| 子状态 | 条件 | 推荐动作 |
|---|---|---|
| `HOLD` | `edge > +2pp` | 持有 |
| `MARGINAL` | `−3pp ≤ edge ≤ +2pp` | 观察 |
| `AT_TARGET` | `edge < −3pp` 且未重评 | 建议清仓 |
| `SOFT_NEGATIVE` | `edge < −3pp` + 重评过 | 重评 / 决定 |

## 六、DISCOVERY Prompt (新仓研究)

### 6.1 触发

Dashboard 点 "🤖 复制给Claude" → `/api/full_prompt` → DISCOVERY 模板 + 最近扫描结果。

### 6.2 关键流程

每个候选要 Claude:
1. **Bull / Bear 视角** (各 1-2 个权威来源)
2. **原始估算 raw**
3. **校准后** = `市场价 + 0.5 × (raw − 市场价)` (五折)
4. **校准后 Edge** = calibrated − cur
5. **执行成本** = taker 滑点 + 双边 2.2pp
6. **实际可获取 Edge** = 校准 edge − 执行成本

### 6.3 推荐门槛

| 模式 | 校准 edge 门槛 |
|---|---|
| 标准 | ≥ **8pp** |
| 中范围 | ≥ **11pp** |
| 大范围 | ≥ **14pp** |

### 6.4 价格区间不对称

| 模式 | 价格范围 |
|---|---|
| 标准 | 8% — 92% |
| 中 | 5% — 95% |
| 大 | 3% — 97% |

低端宽因 < 15% 是 longshot 反向操作甜区。

### 6.5 年化 IRR

```
IRR = (calibrated − p) / p × 365 / days × 100%
< 30%: 不推荐 | 30-100%: 可接受 | > 100%: 甜区
```

### v5.0 (2026-05-16 归档) — 入场分档止损 + 急跌冻结 + 40% 兜底

### v5.4 (2026-05-21) — 触发用价不对称: best_bid for TP, cur for SL

## 十五、Dashboard 公开访问与安全 (v5.7, 2026-05-29)

> 本章独立讲网站层 — 怎么暴露 dashboard, 怎么保证只有自己 (和受邀者) 能操作, 自己用又不至于太麻烦.
> UI 界面解析在 §三, 这里只讲 *网络 + 身份验证 + 凭据管理*.

### 15.1 威胁模型

Dashboard 提供 10 个敏感 POST 端点 (详见 §十), 凡是能调用的人都可以:

| 端点 | 杀伤 |
|---|---|
| `/api/force_exit` | 强制清空仓位 (资金毁灭级) |
| `/api/buy_position` | 强制买入 (烧光现金) |
| `/api/execute_state` | 触发 monitor 决策卖出 |
| `/api/mark_reeval`, `/api/update_q`, `/api/update_tp`, `/api/update_tier` | 篡改后 monitor 心跳会按假数据自动卖 (间接资金杀) |
| `/api/control` | 停止 bot |
| `/api/record_position`, `/api/update_confidence` | meta 污染 |

→ **必须验证身份**, 不能裸露公网.

### 15.2 设计原则

1. **自己用零摩擦** — 本机 (Mac mini) 不需要登录
2. **手机/笔记本用首次登录, 之后 90 天免** — 不能每次都输密码
3. **公网无 URL 可攻击** — 即使有人知道你的项目存在也找不到入口
4. **设计假设公开** — 安全靠 secret 不靠 obscurity (Kerckhoffs)

### 15.3 三层防御架构

```
[外部公网]    ─ ✗ ─→ (无路径; DNS 在 tailnet 外不解析)
[Tailnet 设备] ─→ tailscale serve (本机 daemon, HTTPS 终结)
                     │
                     ▼
                Flask (127.0.0.1:5051, 只 listen localhost)
                     │
                     ├─ 来源判定:
                     │    A. remote_addr ∈ {127.0.0.1, ::1} AND 无 X-Forwarded-For
                     │       → 本机直连, 放行 (零摩擦)
                     │    B. 其他来源 → 走身份验证
                     │
                     ▼
                Flask session 检查
                     │
                     ├─ session.authed=True   → 放行
                     ├─ path=/login           → 放行
                     ├─ method=POST 且未授权  → 401 JSON
                     └─ GET 且未授权          → 302 → /login
```

**关键踩坑** (5-29 实测确认): Tailscale serve 是 daemon 内部反代, **它转发到 Flask 时 `remote_addr` 也是 `127.0.0.1`**. 如果只判 `remote_addr in (127.0.0.1, ::1)` 就放行, **整层鉴权完全失效, tailnet/Funnel 流量被当成本机**. 必须同时判 `X-Forwarded-For` 头存在与否 — Tailscale serve 反代会塞这个头, 本机直连不会有.

### 15.4 Tailscale 层 (基础设施, 一行命令切换)

| 模式 | 命令 | 谁能访问 |
|---|---|---|
| Funnel (公网) | `tailscale funnel --bg on` | 任何拿到 URL 的人 |
| Serve (tailnet-only) | `tailscale serve --bg http://localhost:5051` | 只接入 tailnet 的设备 |

**默认推荐 Serve**. URL 不变 (仍是 `https://<hostname>.tailXXXXXX.ts.net`), 但 DNS / TLS 只对 tailnet 设备 resolve, 公网完全找不到入口.

**临时分享给非 tailnet 朋友**: 临时 `tailscale funnel --bg on`, 朋友凭密码看, 看完 `tailscale funnel --bg off`. 即使 Funnel 开着, Flask 密码层仍挡住攻击.

### 15.5 Flask 密码层 (代码, ~80 行集中在 `modules/dashboard.py`)

**配置块** (放在 `create_app()` 一进来):
```python
app.secret_key = os.environ.get("FLASK_SECRET_KEY") or secrets.token_urlsafe(32)
app.config.update(
    PERMANENT_SESSION_LIFETIME=timedelta(days=90),
    SESSION_COOKIE_HTTPONLY=True,   # JS 偷不到
    SESSION_COOKIE_SAMESITE="Lax",  # CSRF 防御
    SESSION_COOKIE_SECURE=True,     # 只走 HTTPS (Tailscale 都是 HTTPS)
)
```

**中间件** (`@app.before_request`):
```python
is_local_direct = (
    flask_request.remote_addr in ("127.0.0.1", "::1")
    and not flask_request.headers.get("X-Forwarded-For")
)
if is_local_direct:
    return None
if session.get("authed"):
    return None
if flask_request.path in ("/login", "/favicon.ico"):
    return None
if flask_request.method == "POST":
    return jsonify({"ok": False, "error": "auth required"}), 401
return redirect("/login")
```

**登录路由** (`/login`):
- GET → 渲染极简 LOGIN_HTML form
- POST → 比对 `os.environ["DASHBOARD_PASSWORD"]`, 正确则 `session["authed"]=True; session.permanent=True`

**限流** (in-memory dict): 5 次错密码 → 30 分钟 lock by IP. 简单可靠, 不依赖 Flask-Limiter.

### 15.6 端口收紧 (`main.py`)

```python
app.run(host="127.0.0.1", port=5051, debug=False)  # 不是 0.0.0.0
```

### 15.10 安全等级 / 攻击面分析

要攻入 dashboard 需要同时打穿三关:

| 关卡 | 攻击成本 |
|---|---|
| 进 tailnet | 拿到 Tailscale 账号 (邮箱 + 密码 + 2FA), 难度高 |
| Flask 密码 | 暴力破解 32 字符随机, 理论上 ~2^192 次, 不可能 |
| 限流 | 5 次失败 30 分 lock by IP, 自动化爆破成本巨高 |

**剩余攻击面**:
- secret_key 泄漏 → 攻击者可伪造 session (绕过密码). 缓解: secret_key 在 gitignored `.env`, 公开文档绝不写真值.
- Tailscale 账号被偷 → 完全绕过. 缓解: 启用 2FA, 用密码管理器.
- in-memory 限流重启清零 → 小漏洞, 但攻击者要先重启系统才能利用, 实际不可行.

### 15.11 公开本设计的边界

本章可以完整发布 — **架构 / 代码骨架 / 部署步骤公开不影响安全** (Kerckhoffs principle).

**绝对不能写进公开文档的真值** (即使是私有 README 也建议放占位符):

- `DASHBOARD_PASSWORD` 真值
- `FLASK_SECRET_KEY` 真值
- 真实 Tailscale 账号邮箱
- proxy wallet 地址 (公开 = 任何人能查链上完整持仓和交易历史)
- private key (永远不要)
- 真实 tailnet hostname (可保守替换为 `<hostname>.tailXXXXXX.ts.net`)

并发改动**理论上**不改变扫出来的候选集合 (输入 / API / 阈值都没动). 但加 5 层防御保证:

1. **A/B 验证** (实施前必跑): 同 tag 跑 serial 和 parallel, diff 输出 — **实测 0 差异**, 仅时间戳行不同.
2. **显式 sort 输出**: 报告里 markets 按 volume 降序, 不依赖处理顺序.
3. **HTTP retry**: `_s()` 用 `Retry(total=3, backoff_factor=1)` — 旧版本就有, 沿用.
4. **环境变量 fallback**: `.env` 设 `SCAN_PARALLEL=0` → 回串行. `_scan_parallel()` 函数动态读 env, 无需 reload.
5. **报告版本标记**: 顶部 `<!-- scan version: parallel-v1 -->` 注释, 工具可识别版本.

### 18.10 UX 微调: 复制即标记 + 重置按钮 (2026-05-31 同日补丁)

用户反馈: "全扫 27 tag 完成后, 一个个看一个个复制, 但视觉上没法知道哪些已经处理过了, 容易重复粘贴或漏粘贴". 加 2 个交互:

1. **🤖 复制即标记**: `copyP()` 复制成功后, 自动把当前 tag 的 chip 状态清掉 (从 ✓ 绿色变回原样无修饰). 视觉信号 "这个 tag 我已经送 Claude 了".
   - chip 自身仍然存在, 缓存报告仍在 `data/scan_reports/<slug>.md`
   - 用户仍然可以点 chip 切换回该 tag 的报告查看
   - 但视觉上明确区分: ✓ 绿色 = 待复制 / 原样 = 已复制过
2. **🔄 重置标记按钮**: 在 🚀 全扫按钮旁边, 点击把所有 chip 的 ✓ / ⏳ / 🔄 / ❌ 状态全部清掉, 进度文字也清空.
   - **不删缓存文件** — 只清 UI 标记
   - 点 chip 仍能从缓存看报告

JS 实现 (`modules/dashboard.py`):
```js
function doResetScanMarks(){
  const chips = document.querySelectorAll('.tag-chip[data-tag]');
  chips.forEach(chip => {
    chip.classList.remove('chip-pending','chip-running','chip-done','chip-error');
    chip.textContent = chip.getAttribute('data-tag');
  });
  // 进度文字也清空
}

// copyP() 成功后:
if (cur) { setChipStatus(slugifyTag(cur), ''); }  // empty status = 移除 emoji 和 class
```

`setChipStatus` 已经支持空 status 参数, 不需要新代码 — emojiMap['undefined' or ''] 为 undefined, `undefined||''` 是 '', `textContent = label + ''` 等于干净 label. 复用现有 helper.
